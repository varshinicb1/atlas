failures:
  - id: failure:astro/hydration-without-island
    symptom: Interactive component renders static HTML with no JavaScript; clicks do nothing.
    cause: Using a framework component (React/Vue) without a client:* directive; Astro ships no JS by default.
    fix: Add client:load, client:idle, client:visible, or client:media to the component tag.
  - id: failure:astro/glob-pattern-wrong
    symptom: Astro.glob returns an empty array or missing files at runtime.
    cause: The glob pattern does not match the actual file location; Astro.glob uses static analysis and only supports literal string patterns.
    fix: Verify the pattern path relative to the current file; use a literal string (not a variable) and ensure files exist.
  - id: failure:astro/ssg-dynamic-route-404
    symptom: getStaticPaths not called or dynamic route returns 404 at build.
    cause: Missing or incorrectly exported getStaticPaths in an SSG route; Astro requires it to pre-render dynamic pages.
    fix: Export async function getStaticPaths() returning an array of { params } objects matching the file's bracket params.
  - id: failure:astro/ssr-cookie-not-set
    symptom: Cookies set via Astro.cookies.set are not reflected in the browser on first load.
    cause: Setting a cookie after the response headers have already been sent; cookies must be set before returning the response.
    fix: Set cookies in middleware or early in the page script before any rendering; use context.cookies.set in middleware.
  - id: failure:astro/content-schema-mismatch
    symptom: "ZodError: Validation failed for entry X" at build time.
    cause: Frontmatter in a content entry does not match the defineCollection schema; required fields are missing or have wrong types.
    fix: Update the markdown frontmatter to match the schema or adjust the schema to make the field optional (z.string().optional()).
  - id: failure:astro/adapter-missing
    symptom: "Astro couldn't find an adapter" error when using output: 'server' or output: 'hybrid'.
    cause: SSR mode requires an adapter (node, vercel, cloudflare, netlify) installed and configured in astro.config.
    fix: Run npx astro add <adapter> or manually install and add the adapter to astro.config.mjs.

# Astro.js Web Framework

Astro is a web framework for building content-driven websites with zero JavaScript by default. Every `.astro` component runs its frontmatter script at build or request time, strips the runtime JavaScript, and emits static HTML. Interactive UI is opt-in via **islands**: framework components (React, Vue, Svelte, Preact, Solid, Lit) that hydrate in isolation with `client:*` directives. This architecture delivers fast initial loads because the browser parses minimal JS, and only the interactive parts pay the hydration cost.

Content management is built around **content collections** — `src/content/` directories with Zod-schema-validated frontmatter. `getCollection('posts')` returns typed entries with inferred TypeScript types, caught at build time rather than runtime. For dynamic needs, Astro supports **SSR** (`output: 'server'`) and **hybrid** mode where each route opts in to per-request rendering or static generation via `export const prerender`. SSR enables middleware chains, cookies, API endpoints, and direct database access through **Astro DB**, which wraps LibSQL with typed table definitions and SQL queries.

The **integration** ecosystem extends Astro without custom Vite config. Framework adapters, sitemaps, RSS feeds, image optimization (`astro:assets`), and analytics are all one `npx astro add` away. The **Image** component from `astro:assets` automatically generates responsive `srcset` attributes, converts to modern formats (AVIF, WebP), and applies lazy loading — for both local and remote images. **View Transitions** provide SPA-like animated navigations without a client-side router by leveraging the browser's View Transitions API, controlled through the `ViewTransitions` component and `transition:name` / `transition:animate` directives on elements.
