    description: "Type-safe relation declarations — relations(users, ({ hasMany }) => ({ posts: hasMany(posts, { from: users.id, to: posts.authorId }) })). Supports hasOne, hasMany, belongsTo, manyToMany via junction tables. Relations power the relational query API."

failures:
  - id: failure:drizzle/n-plus-one-without-relations
    symptom: Multiple database queries when accessing related data in a loop.
    cause: Using raw db.select for related data inside a forEach instead of using the relational query API.
    fix: "Use db.query.users.findMany({ with: { posts: true } }) for eager loading. For more complex joins, use db.select().from(users).leftJoin(posts, eq(users.id, posts.authorId))."
  - id: failure:drizzle/schema-dialect-mismatch
    symptom: SQL syntax errors when using PostgreSQL-specific features on MySQL.
    cause: Using pgTable with MySQL database, or using MySQL-specific column types with PostgreSQL.
    fix: Import the correct table builder for your database (pgTable, mysqlTable, sqliteTable). Check column type compatibility — jsonb is PostgreSQL-only, text is universal.
  - id: failure:drizzle/migration-conflicts
    symptom: Migration apply fails with "already exists" or column conflicts.
    cause: Multiple developers modifying schema and generating migrations independently on the same branch.
    fix: Always pull latest migrations before running drizzle-kit generate. The snapshot file tracks the current state — conflicts appear as snapshot mismatches. Resolve by regenerating from the latest main branch snapshot.
  - id: failure:drizzle/type-unsafe-raw
    symptom: Runtime errors from mismatched SQL types when using sql template literal.
    cause: "Using sql\\`${value}\\` without proper parameterization, or mixing SQL fragments from different dialects."
    fix: "Always pass values as placeholders (sql\\`SELECT * FROM users WHERE id = ${id}\\`) — Drizzle parameterizes automatically. For dynamic identifiers, use sql.identifier()."

---
Drizzle's philosophy is that ORMs should not hide SQL — they should enhance it with type safety. Unlike Prisma's generated client that abstracts SQL behind a proprietary API, Drizzle's query builder mirrors SQL structure directly: SELECT becomes db.select(), WHERE becomes .where(eq(...)), JOIN becomes .leftJoin(). This means developers who know SQL can write Drizzle queries immediately, and the generated SQL is predictable and readable.

The dual query API is Drizzle's distinguishing feature: the SQL-like builder for maximum control (db.select().from(users).leftJoin(posts, ...)) and the relational API (db.query.users.findMany({ with: { posts: true } })) for convenience. The relational API is powered by the relations() declarations in the schema, which define the data model graph. A single relational query generates an optimized SQL JOIN — no N+1, no lazy loading surprises.

Drizzle Kit's migration system generates plain SQL files — not binary blobs or proprietary formats. Each migration is a timestamped pair of .sql files (up and down) that you can read, review in PRs, and edit if needed. The snapshot.json file tracks the current schema state for differential migration generation. drizzle-kit push is the fast dev loop; drizzle-kit generate + migrate is the production workflow.
